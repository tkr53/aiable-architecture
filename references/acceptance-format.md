# acceptance.md の書式

`acceptance.md` は capability の受け入れ条件を定める一次資料である。test/ はこのファイルから
生成され、各テストは末尾の trace 表で受け入れ条件 (AC) と 1:1 で対応する。このファイルの
目的は、人間がテストコードを読まずに「受け入れ条件に過不足がないか」だけを見て承認できる
状態を作ることにある。

## 設計の四要件

このファイルが満たすべきことは四つある。第一に、各 AC に一意の ID を振り、test と機械的に
1:1 対応が取れること。第二に、各 AC が「例示で確認するもの (example)」か「プロパティで確認
するもの (property)」かを明示分類できること。第三に、人間が承認時に読むのはこのファイルだけ
で完結し、生成されたテストコードを読まずに済むこと。第四に、AI がこのファイルから決定的に
テストを生成でき、逆にテストからこのファイルへ追跡し返せること。

これを満たすため、各 AC は「人間が読む散文」と「機械が読む構造ブロック」の二層を持つ。
散文だけだと変換が非決定的になり、構造だけだと人間の承認が形骸化する。二層に分け、散文は
「なぜ」を、構造は「何を」を担当させ、担当を重ねないことでドリフトを防ぐ。

## 散文と構造の分離（最重要のドリフト対策）

同じ事実を二度書かない。given/when/then や examples 表のような「機械が使う値」は構造ブロック
にしか書かない。散文側（見出しと intent 行）には値を書かず、なぜその条件が要るのかという
意図だけを書く。散文に値が混入していたら red flag であり、値は構造へ移す。

構造ブロックは `<!-- normative:begin AC-xxx -->` と `<!-- normative:end AC-xxx -->` で囲む。
囲みの内側が「意味を持つ＝変われば再承認」の領域、囲みの外（intent や見出しの散文）が
「補助＝変わっても再承認不要」の領域である。AI がテストを生成する際、変換の根拠にしてよいのは
この囲みの内側だけである。intent は読んでよいが、そこからテストを導出してはならない。

## AC の書式

各 AC は次の形を取る。

```markdown
### AC-001  既存メールでは登録できない
**intent:** 同一 email の二重登録を防ぐ。データ整合性の根幹。
            大小・前後空白の正規化後に判定する。正規化前に比較すると重複をすり抜けるため、
            正規化後の比較であることをここで意図として明示する。
**kind:** example

<!-- normative:begin AC-001 -->
- given: email "a@example.com" が既に登録済み
- when:  正規化後に同一となる email で登録を試みる
- then:  登録は失敗し、理由は EMAIL_TAKEN
- and:   users ストアの件数は増えない

examples:
| email          | 事前状態 | 期待結果     |
|----------------|---------|-------------|
| a@example.com  | 登録済み | EMAIL_TAKEN |
| A@Example.com  | 登録済み | EMAIL_TAKEN |
|  a@example.com | 登録済み | EMAIL_TAKEN |
<!-- normative:end AC-001 -->
```

property の場合は kind を property とし、normative ブロックに property-type・generator・
不変条件を書く。型ごとの書式は `property-types.md` に従う。

```markdown
### AC-002  登録は件数を保存する
**intent:** どんな入力列を与えても、成功した登録回数とストア件数が一致する。失敗は無影響。
**kind:** property

<!-- normative:begin AC-002 -->
- property-type: 不変量 (invariant)
- generator: 有効/無効が混在する登録リクエストの列（長さ 0〜N）
- invariant: users.count == 成功した登録の件数
- note: 失敗リクエスト (EMAIL_TAKEN, VALIDATION_ERROR) はストアを変更しない
<!-- normative:end AC-002 -->
```

## 未解決質問セクション（spec フェーズの停止条件の片方）

ファイル冒頭に未解決質問セクションを置く。spec フェーズ中に AI が出した疑問をここに構造化して
保持し、議論で解消するたびに消し込む。このセクションは normative の外（散文側・再承認対象外）
にあるが、承認の門番として機能する。

```markdown
## 未解決質問
（空であること。残っている限り承認に進めない）
- [ ] 例: パスワードの最小長は仕様で定めるか、別 capability に委ねるか
```

spec フェーズの完了条件は厳格に二つで、両方を満たさねばならない。第一に、全 AC に normative
ブロックが揃っていること（下記 lint で検査）。第二に、この未解決質問セクションが空であること。
片方でも欠ければ議論を続ける。

## trace 表（承認の物理的な核）

ファイル末尾に trace 表を置く。人間はテストコードを読まず、この表の行数と各行の AC-ID を見て
過不足を判断する。表にあって AC 節がない、AC があって行がない、はいずれも不整合である。

```markdown
## trace
| AC-ID  | kind     | test 名                                    | 状態   |
|--------|----------|-------------------------------------------|--------|
| AC-001 | example  | TestDuplicateEmailRejected                | 凍結   |
| AC-002 | property | TestCountInvariant                        | 凍結   |
```

行の増減ができるのは人間だけである。AI は行を増減できない。これは「凍結テストは AI が編集
できない」を表のレベルで言い換えたものである。

## 網羅性チェック（承認の形骸化を防ぐ最後の仕掛け）

trace 表の下に、人間が承認時に答えるチェックリストを置く。

```markdown
## 網羅性チェック
- [ ] 失敗系は全て列挙したか（拒否理由に漏れはないか）
- [ ] 各 property の generator は「実装で対応予定の入力」より広いか
- [ ] impl を見て後から書き足した条件はないか（あれば red flag）
```

最後の項目はオラクル問題の再発を自己点検するための問いである。テストや条件を実装に合わせて
書いていないか、人間が定期的に確認する。

## 第二層 lint（今すぐ入れる）

acceptance.md 自体に対する存在検査。テスト生成の前段で走らせる。検査内容は次の通り。

- AC 節があるのに normative ブロックが空でないか
- kind が property なのに normative 内に property-type / generator が無いか
- AC-ID が見出しと trace 表で食い違っていないか
- trace 表に対応する AC 節が存在するか
- 未解決質問セクションが承認時点で空か

これらは値の一致ではなく存在レベルの整合なので機械的に検出できる。値の一致検査は不要である
（散文と構造で値を二重に書かない方針により、そもそも食い違う値が存在しないため）。

## 第三層 ハッシュ承認（後付け）

承認の形骸化と「いつのまにか書き換わる」を防ぐ最終手段。承認時点の acceptance.md の
**normative ブロックの内側を連結したもの**のハッシュを記録し、テスト生成はそのハッシュに対して
行われたという結線を残す。normative の外（intent 等の散文）はハッシュ対象外なので、文言調整
では再承認が走らず、given/when/then や examples 表や property 定義が変わったときだけ再承認が
要求される。

この機能は最初から入れず、フォーマットと生成フローが安定してから後付けする。後付けを純粋な
追加にするための仕込みが、上記の `normative:begin/end` 区切りである。区切りの内側を連結して
ハッシュを取るコードを足すだけで、フォーマットには一切触れずに承認固定が乗る。
