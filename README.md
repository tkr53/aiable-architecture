# Aiable Architecture

Software architecture for the era of generative AI. It is designed on the premise that
**AI writes the implementation, and humans own only the spec and the tests**.

> 日本語版は [README.ja.md](README.ja.md) を参照。

## First principle

Source code is an artifact that can be regenerated from the spec and the tests; the assets
truly worth preserving are the three of **spec, test, and incident**. The implementation
(impl) is disposable and can be rebuilt at any time. The entire architecture is deduced from
this inversion.

## The five pillars

1. **Humans do not write impl directly.** Humans touch only spec and test; the implementation
   is generated by AI.
2. **Split by capability, not by layer.** Keep the spec/test/impl/incident of a single concern
   physically adjacent, minimizing the context the AI must load.
3. **Make tests the anchor of trust.** But tests must be traceable to the spec. Tests are
   approved and frozen before the implementation exists.
4. **Discuss the spec exhaustively and minimize questions and uncertainty.** The spec phase has
   a strict stop condition (every AC has a normative block, and the open-questions list is empty).
5. **Give every capability a spec / test / incident.** An incident records the cause of a
   failure and its countermeasure, and is always wired to a regression test so that the same
   problem is never hit twice — guaranteed by a test.

## Trust model

Tests and properties are generated by AI, and humans judge their completeness on `acceptance.md`
and approve them. Approval happens before the implementation, and the tests are frozen. Once
frozen, the AI cannot edit the tests (to prevent loosening tests just to make the implementation
green). If the AI believes a test is wrong, it does not fix it on its own but escalates to a
human. The authority to move the verification always rests with the spec layer (the human).

## Test strategy: three layers

- **Example-based** — the concrete inputs and outputs the human has in mind.
- **Property-based (PBT)** — declare the input space and let the machine explore it. Uses
  `pgregory.net/rapid`.
- **Mutation testing** — confirm, before approval, that the tests actually bite.

## Directory layout

```
capabilities/
  <capability>/
    spec/        intent.md, acceptance.md (normative blocks, trace table, open questions)
    test/        the frozen anchor of trust. Generated from acceptance.md
    impl/        AI-owned, disposable
    incident/    failure records. Wired to regression tests
    contract.md  the promise to the outside (the only thing referenced between capabilities)
shared/          minimal. Purely technical elements with no behavior only
```

## Install (Claude Code plugin)

This repository is a Claude Code plugin marketplace. Add the marketplace and install the plugin:

```
/plugin marketplace add tkr53/aiable-architecture
/plugin install aiable-architecture@aiable-architecture
```

This installs the `aiable-architecture` skill (which activates automatically by context) and the four
`/aiable-*` commands below.

## How to use this Skill

`skills/aiable-architecture/SKILL.md` is the body of the architectural discipline.
`skills/aiable-architecture/references/` holds the formats and conversion templates.

- `skills/aiable-architecture/references/acceptance-format.md` — the format of acceptance.md
- `skills/aiable-architecture/references/property-types.md` — converting the 5 property types into rapid
- `skills/aiable-architecture/references/incident-format.md` — the incident format and wiring to regression tests

Command entry points (in Claude Code as `/<command-name> <capability>`, or via autonomous
invocation):

- `/aiable-spec` — nail down the spec
- `/aiable-test` — generate, approve, and freeze the tests
- `/aiable-impl` — generate the implementation that satisfies the frozen tests
- `/aiable-incident` — record a failure and wire it to a regression test

## Worked example

`capabilities/user-registration/` is a model that conforms to every convention. To run it locally:

```bash
cd capabilities/user-registration
go mod tidy
go test ./...
```

The example tests use only the standard library; the property tests use rapid. With `go test -fuzz`
you can run the exploration jobs (FuzzCountInvariant etc.) on a separate track.

## Status

This is a reference implementation of the design — a model meant to convey the philosophy. When
applying it to a real project, reinterpret the language, test infrastructure, and CI setup to fit
your own environment.
