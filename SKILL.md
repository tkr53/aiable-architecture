# Aiable Architecture

これは「AI が実装を書き、人間は仕様とテストだけを所有する」ことを前提に設計された
ソフトウェアアーキテクチャの規律である。このファイルは、その前提のもとで作業する際に
常に従うべき第一原理・信頼モデル・物理構成・正準フロー・テスト戦略・incident 運用・
コマンド入口を定める。個別の書式や変換規則は `references/` に分離してあり、必要な
フェーズに入ったときに該当ファイルを読む。

このアーキテクチャの全体像を一文で言うと、**ソースコードは spec とテストから再生成可能な
生成物であり、真に保存すべき資産は spec・test・incident の三点である**、という主張に尽きる。
以降の規律はほぼ全てこの一点から演繹される。なぜそうするのかを理解した上で運用してほしい。
理由を理解していれば、ここに明記のない状況でも正しい側に倒せるからである。

---

## 1. 第一原理：コードは生成物、spec/test/incident が source

従来のアーキテクチャはソースコードを一次資産として扱い、テストや仕様をその補助物として
扱う。Aiable Architecture はこれを逆転させる。実装（impl）は spec を満たすために AI が
生成する使い捨ての成果物であり、いつでも捨てて再生成できる。真の source は次の三つである。

- **spec** — 何を満たすべきか。人間が議論して確定させる一次資料。
- **test** — spec が満たされていることを自動で検証する、信頼の錨。
- **incident** — 過去に踏んだ失敗とその恒久対策。再発を防ぐための学習資産。

この逆転を受け入れると、いくつかの帰結が自動的に従う。人間は impl を直接編集しない
（編集すると再生成性が壊れ、spec と impl がドリフトする）。impl のコード量や内部構造の
美しさそのものは資産ではない。spec と test が正しければ、impl は何度でも作り直せる。

> なぜ逆転させるのか。AI が実装を担うなら、人間の関与は最も価値の高い場所に集中させる
> べきだからである。最も価値が高いのは「何を作るべきか」の決定（spec）と「正しさをどう
> 保証するか」の決定（test）であって、「どう書くか」（impl）ではない。後者は AI に委ねる。

## 2. 信頼モデル：AI 生成・人間承認、テストは実装より先に凍結する

このアーキテクチャの心臓は、信頼の錨を必ず AI の外側に打つことである。AI が実装を書き、
AI がテストも書くなら、両方が緑でも何も証明されない（実装に都合よくテストを書いた可能性が
あるため）。これを避けるための規律が以下である。

**作者と承認者の分離。** テストとプロパティは AI が生成してよいが、その過不足を人間が
判断して承認する。承認の対象は生成されたテストコードではなく、`acceptance.md` に人間語で
並んだ受け入れ条件とプロパティの一覧である。人間はテストコードを読まず、受け入れ条件の
リストに抜けや過剰がないかを見る。

**実装より先にテストを凍結する。** これが信頼モデルで最も重要な順序である。実装が先に
存在すると、AI の書くテストはその実装を緑にするよう逆算され、人間の承認は「テストが仕様を
捉えているか」ではなく「テストが実装と一致しているか」の確認に堕する。それは何も証明しない。
だからテストは、実装が存在しない段階で、spec だけを根拠に承認・凍結する。後から実装が
それを緑にした瞬間に、初めて緑が意味を持つ。承認が実装から独立していることが、信頼を
外側に錨で留める条件そのものである。

**凍結後の権限境界。** 承認・凍結されたテストとプロパティを、AI は編集できない。実装を
緑にするためにテストを緩めるのは最悪の失敗モードだからである。AI がテストを間違っていると
判断した場合、勝手に直さず「これは仕様の問題ではないか」と人間にエスカレーションする。
凍結された検証に対して AI ができるのは編集ではなく異議申し立てだけであり、検証を動かす
権限は常に仕様レイヤー（＝人間）にある。PBT が反例を見つけた場合も同じで、AI がプロパティを
弱めたり生成範囲を狭めたりして緑にするのは禁止である。反例は「実装のバグ」か「プロパティの
誤り」のどちらかであり、後者だと思うなら人間に上げる。

> 承認の形骸化を防ぐ仕掛けは `references/acceptance-format.md` に定める。要点は、人間が
> 読むのは acceptance.md だけで足りるよう、受け入れ条件とテストを 1:1 でトレースさせること。

## 3. 物理構成：capability で割る、層で割らない

ディレクトリは技術レイヤー（controller / service / repository など）で割らない。層で割ると
一つの機能が複数ディレクトリに散り、AI が一箇所を直すために読み込むコンテキストが膨らむ。
代わりに **capability（一つの仕様として閉じた振る舞いのまとまり）** で割り、その内部に
spec・test・impl・incident・contract を隣接させる。一つの関心事に必要なものが物理的に
一箇所に集まるので、AI はそのスライスだけをコンテキストに載せれば作業が完結する。

```
capabilities/
  <capability-name>/
    spec/
      intent.md       # 目的・背景・非機能要件・トレードオフの決定記録・未解決質問
      acceptance.md   # 受け入れ条件（test と 1:1）。normative ブロックを持つ
    test/             # 凍結される信頼の錨。acceptance.md から生成・トレース可能
    impl/             # AI 所有・使い捨て・再生成可能
    incident/         # 障害の原因と恒久対策。回帰テストへ結線される
    contract.md       # この capability が外に約束する入出力の境界
shared/               # 最小限。振る舞いを持たない技術的要素のみ
```

**capability 間の参照は contract 経由の一方向のみ。** ある capability が別の capability に
依存するとき、参照してよいのは相手の `contract.md` だけであり、相手の impl を参照しては
ならない。impl は使い捨てなので、impl に依存すると再生成のたびに依存先が壊れる。contract
だけが capability 間の安定した約束である。

**shared は痩せさせる。** shared に置けるのは、どの capability の仕様にも属さない、純粋に
技術的で振る舞いを持たない要素（横断的な型定義、フォーマット定数など）に限る。判定は
「これは spec/test を持つべきか」で行う。持つべきなら、それは振る舞いであり、必ずどこかの
capability に属させる。shared に逃がしてよいのは、テストする必要のないものだけである。

> なぜここまで shared を絞るのか。shared は「とりあえずここに置く」誘惑が働く場所であり、
> 一度緩めると局所性が崩れて、AI が一機能を直すために shared 全体を読む羽目になるからである。

## 4. 正準フローと各フェーズの完了条件

capability は次の順序で構築・保守される。各フェーズには明確な完了条件があり、満たすまでは
次に進まない。

1. **spec を詰める。** intent.md と acceptance.md を人間と AI が議論して確定させる。
   疑問や不確定要素を徹底的に潰す。
   **完了条件（厳格・両方必須）**：(a) 全ての受け入れ条件に normative ブロックが揃っている
   （`references/acceptance-format.md` の lint で検査可能）、かつ (b) intent.md の未解決質問
   セクションが空である。両方を満たすまで承認に進まない。片方でも欠ければ議論を続ける。

2. **test を生成・承認・凍結する。** 凍結済みの acceptance.md から、AI がテスト三層を生成する。
   人間が acceptance.md 上で過不足を判断して承認し、テストを凍結する。詳細は §5。
   **完了条件**：trace 表の全行が「凍結」状態であり、人間の承認が得られている。

3. **impl を生成する。** 凍結テストが全て緑になるまで AI が実装を生成する。impl は使い捨て。
   **完了条件**：凍結テスト三層が全て緑。テストを緩める形での緑化は禁止（§2 の権限境界）。

4. **incident を運用する。** 本番で障害が出たら incident に記録し、回帰テストへ結線して
   acceptance.md にプロパティ／例示を追加し、フェーズ2に戻る。詳細は §6。

> フェーズ1の停止条件を機械的に観測可能にしているのは、原則「疑問を最低化する」を
> 人間の感覚任せにしないためである。停止条件がファイル上の状態（lint 通過・質問セクションが空）
> として表現されていれば、「もう十分」が主観でなく検査になる。

## 5. テスト戦略：例示・プロパティ・ミューテーションの三層

テストは三層で構成し、それぞれ役割が異なる。三層とも acceptance.md を唯一の根拠とし、
実装から導出してはならない（実装から導くとオラクル問題が再発する）。

- **例示ベース（example）** — 人間が想定する具体的な入出力を確認する。acceptance.md の
  `kind: example` の条件に対応する。
- **プロパティベース（property, PBT）** — 入力空間そのものを宣言して機械に探索させ、
  人間が想像しにくい入力（境界・空・極大・Unicode・順序崩れ）での不変条件を検証する。
  `kind: property` の条件に対応する。プロパティは人間が acceptance.md に言葉で書き、AI は
  それを生成器とアサーションに変換するだけ。プロパティの型（往復性・不変量・冪等性・
  順序非依存・対オラクル）は `references/property-types.md` の語彙に従う。
- **ミューテーション検査** — 上記テストが「ちゃんと噛む」かを承認前に確認する手段。AI に
  わざと間違った実装を数個出させ、凍結予定のテストがそれを赤にできるかを見る。テストが
  緩くてどんな実装でも通る、という見逃しを防ぐ。

役割の整理：ミューテーションは**テストの強度**を検査し（テストを疑う）、PBT は**仕様の網羅性**
を検査する（実装を疑う）。両者は補完関係にある。

**PBT の運用上の制約。** 反例は確率的に出るため、凍結テストは固定シードで決定的に再現
できなければならない（実行ごとに通ったり落ちたりするものは信頼の錨として失格）。CI では
固定シードと探索回数を明示し、別枠で乱択シードの探索ジョブを回して、新しい反例が出たら
incident 化する。反例を見つけたら、最小化された入力を incident に紐づく例示ベースの回帰
テストへ蒸留して固定する。

詳細な変換テンプレート（各 property 型をどうテストコードに落とすか）は
`references/property-types.md` に置く。テスト生成フェーズでそれを読む。

## 6. incident 運用：失敗を必ず回帰テストへ結線する

incident は「自分の失敗に対する検索可能な記憶」である。人間向けのポエムではなく、AI が
関連作業時にコンテキストへ載せられる構造化された記録にする。各 incident は最低限、症状・
再現条件・根本原因・恒久対策・**それを守る回帰テストの名前**を持つ。

最後の項目が本質である。incident が必ず一本の回帰テストに結線されていれば、「同じ問題を
二度踏まない」がテストによって保証される。incident → test の逆流（失敗を新しいテストに
変えて凍結する）を仕組みとして強制する。詳細な書式は `references/incident-format.md` に置く。

## 7. コマンド入口

利用者は各フェーズを明示的なコマンドで駆動できる。コマンドは入口に過ぎず、規律の本体は
この SKILL.md と references にある。各コマンドは以下の規約に従って動作する。

- **/aiable-spec `<capability>`** — フェーズ1。intent.md と acceptance.md の議論を開始・継続し、
  §4 の厳格な停止条件を満たすまで疑問を潰す。停止条件に達したら承認を促す。
- **/aiable-test `<capability>`** — フェーズ2。凍結済み acceptance.md からテスト三層を生成し、
  ミューテーション検査を通し、人間の承認・凍結に回す。凍結後のテスト編集はしない（§2）。
- **/aiable-impl `<capability>`** — フェーズ3。凍結テストを緑にする impl を生成する。テストを
  緩めない。テストが誤っていると思えば異議を上げる（§2）。
- **/aiable-incident `<capability>`** — フェーズ4。障害を記録し、回帰テストへ結線し、
  acceptance.md に条件を追加してフェーズ2に戻す。

> いずれのコマンドも、§2 の権限境界を破ってはならない。コマンドは作業の開始点を与えるだけで、
> 凍結・承認・エスカレーションの規律はコマンドの外（この SKILL.md）が常に保持する。
